## متن نهایی مقاله

عبارت «API رایگان هوش مصنوعی» در نگاه اول ساده به نظر می‌رسد، اما سرویس‌های مختلف تعریف یکسانی از رایگان ندارند. یک Provider ممکن است سهمیه دائمی محدود ارائه کند، دیگری فقط اعتبار اولیه بدهد، سرویس دیگری پس از پایان Trial نیازمند پرداخت باشد و بعضی سرویس‌ها نیز به‌دلیل سیاست‌های منطقه‌ای یا محدودیت‌های ثبت‌نام برای همه کاربران قابل استفاده نباشند.

به همین دلیل، انتخاب یک API رایگان LLM نباید فقط بر اساس نام مدل یا جمله «Free API» انجام شود. قبل از Integration باید چند سؤال مشخص را پاسخ دهیم.

## ۱. Free Tier، Trial و Credit یکی نیستند

**Free Tier دائمی** یعنی تا زمانی که سیاست سرویس تغییر نکرده، یک سهمیه محدود اما تکرارشونده در اختیار کاربر قرار می‌گیرد. این سهمیه ممکن است روزانه یا ماهانه Reset شود.

**Trial** معمولاً زمان‌دار است. ممکن است چند روز یا چند هفته فعال باشد و پس از آن پایان یابد.

**Credit اولیه** مبلغ یا تعداد Token محدودی است که فقط یک‌بار به Account جدید داده می‌شود. پس از مصرف یا انقضا، ادامه استفاده نیازمند پرداخت است.

همچنین بعضی Gatewayها تعدادی مدل رایگان ارائه می‌کنند، اما محدودیت آن‌ها ممکن است بر اساس مدل، ترافیک یا سیاست Fair use تغییر کند. بنابراین برچسب «رایگان» به‌تنهایی برای تصمیم فنی کافی نیست.

## ۲. آیا کارت بانکی یا روش پرداخت لازم است؟

برخی سرویس‌ها اجازه ساخت API key رایگان را بدون ثبت کارت می‌دهند. بعضی دیگر برای فعال‌سازی Trial یا جلوگیری از سوءاستفاده، Payment method می‌خواهند؛ حتی اگر در همان لحظه هزینه‌ای دریافت نشود.

برای کاربران ایران این موضوع اهمیت بیشتری دارد، چون ثبت کارت بین‌المللی یا تأیید هویت ممکن است عملاً مسیر استفاده را مسدود کند. در نتیجه، باید «رایگان بودن سهمیه» و «امکان ایجاد Account» را دو متغیر مستقل در نظر گرفت.

## ۳. محدودیت واقعی مصرف چیست؟

محدودیت API فقط تعداد درخواست نیست. چند شاخص رایج عبارت‌اند از:

- **RPM:** تعداد Request در دقیقه
- **RPD:** تعداد Request در روز
- **TPM:** تعداد Token در دقیقه
- **TPD:** تعداد Token در روز
- سقف Context window
- حداکثر Output
- محدودیت هم‌زمانی

ممکن است یک سرویس RPD بالایی داشته باشد، اما TPM آن برای پردازش‌های سنگین کافی نباشد. برای Chatbot سبک، محدودیت Request مهم‌تر است؛ برای خلاصه‌سازی اسناد طولانی یا Agentها، Token limit و Context اهمیت بیشتری پیدا می‌کند.

## ۴. سازگاری با OpenAI SDK چه مزیتی دارد؟

بسیاری از Providerها Endpoint سازگار با OpenAI ارائه می‌کنند. در این حالت معمولاً می‌توان با تغییر `base_url`، `api_key` و نام مدل، همان Client موجود را استفاده کرد.

این سازگاری هزینه مهاجرت را کم می‌کند، اما نباید فرض کنیم همه قابلیت‌ها دقیقاً یکسان‌اند. Streaming، Tool calling، JSON mode، Embedding و Error format ممکن است بین Providerها تفاوت داشته باشند. قبل از استفاده در Production، مسیرهای حیاتی باید با تست واقعی بررسی شوند.

## ۵. وضعیت دسترسی از ایران را چگونه بررسی کنیم؟

این بخش بیشترین احتمال خطا و ادعای نادرست را دارد. چند نوع Evidence باید از هم جدا بمانند:

1. سیاست رسمی Provider درباره Regionها
2. مشاهده مستقیم از شبکه داخل ایران
3. مشاهده با VPN
4. نتیجه ثبت‌نام یا Credential validation
5. وضعیت نامشخص یا شواهد متعارض

موفقیت یک Request از Server خارجی ثابت نمی‌کند که سرویس مستقیماً از ایران قابل استفاده است. همچنین نتیجه VPN نباید به‌عنوان دسترسی مستقیم ثبت شود. اگر شواهد کافی وجود ندارد، وضعیت صحیح «نامشخص» است، نه «قابل استفاده».

## ۶. داده‌ها چقدر تازه‌اند؟

Free Tierها سریع تغییر می‌کنند. مدلی حذف می‌شود، سهمیه کاهش می‌یابد، Region جدیدی محدود می‌شود یا Endpoint تغییر می‌کند. بنابراین هر فهرست قابل‌اعتماد باید تاریخ آخرین بررسی، منبع رسمی و سازوکار گزارش داده قدیمی داشته باشد.

یک جدول زیبا بدون تاریخ و Source ممکن است در روز انتشار مفید باشد، اما چند ماه بعد به تصمیم اشتباه منجر شود.

## ۷. یک چک‌لیست ساده قبل از انتخاب Provider

پیش از ساخت Account یا نوشتن Integration، این موارد را بررسی کنید:

- نوع رایگان بودن: دائمی، Trial، Credit یا مدل رایگان
- تاریخ انقضا و نحوه Reset سهمیه
- نیاز به کارت، تلفن یا KYC
- RPM، RPD، TPM و سقف Context
- OpenAI compatibility و Base URL
- مدل‌های موجود در Tier رایگان
- سیاست استفاده از داده و Prompt
- Region policy و نوع Evidence دسترسی
- تاریخ آخرین بررسی
- مستندات رسمی و مسیر گزارش تغییرات
