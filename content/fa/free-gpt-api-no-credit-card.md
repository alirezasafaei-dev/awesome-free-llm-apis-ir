# API رایگان GPT بدون کارت بانکی؛ راهنمای انتخاب امن و واقعی

**پاسخ سریع:** برخی Providerها طبق شرایط جاری، برای Free Tier یا مدل‌های رایگان کارت بانکی نمی‌خواهند؛ اما عبارت «بدون کارت» باید با منبع رسمی یا تجربه تاریخ‌دار پشتیبانی شود. نبود کارت به معنی نبود ثبت‌نام، محدودیت هویت، شماره تلفن، سهمیه پایین یا تغییر آینده نیست. قبل از ساخت پروژه، مسیر Signup تا Inference واقعی را کامل بررسی کنید.

فهرست تاریخ‌دار Providerها و مقدار `requires_payment_method` در [کاتالوگ APIهای رایگان LLM](https://llm.persiantoolbox.ir/) در دسترس است. منبع داده، Schema و فرم گزارش در [مخزن GitHub پروژه](https://github.com/alirezasafaei-dev/awesome-free-llm-apis-ir) قرار دارد.

## منظور از GPT API چیست؟

در جست‌وجوی عمومی، عبارت GPT API گاهی برای هر API چت مبتنی بر مدل زبانی استفاده می‌شود. از نظر فنی باید تفاوت را حفظ کرد:

- API رسمی مدل‌های GPT؛
- Provider سازگار با OpenAI که مدل دیگری ارائه می‌کند؛
- Gateway چندمدلی؛
- مدل متن‌باز با رابط مشابه؛
- سرویس Community یا ناشناس؛
- Trial یا Credit محدود.

اگر پروژه به مدل مشخص GPT نیاز دارد، وجود رابط مشابه کافی نیست. اگر فقط یک API چت سازگار می‌خواهید، گزینه‌های بیشتری وجود دارد. عنوان و مستندات Provider را دقیق بخوانید تا مدل و مالک زیرساخت اشتباه معرفی نشوند.

## «بدون کارت بانکی» چه چیزی را ثابت می‌کند؟

این عبارت فقط باید نشان دهد برای فعال‌کردن مسیر موردنظر، Payment Method لازم نیست. هنوز ممکن است این شروط وجود داشته باشد:

- حساب ایمیل؛
- تأیید شماره تلفن؛
- حساب GitHub یا Cloud؛
- CAPTCHA؛
- محدودیت کشور؛
- دعوت‌نامه؛
- تأیید هویت؛
- Discord signup؛
- Check-in دوره‌ای؛
- مدل‌های محدود؛
- سقف درخواست پایین.

همچنین Provider می‌تواند بعداً Billing details را الزامی کند. بنابراین تاریخ بررسی و منبع رسمی بخش مهم ادعاست.

## انواع دسترسی بدون کارت

### Free Tier حساب‌محور

کاربر حساب می‌سازد و سهمیه رایگان دریافت می‌کند. این مدل معمولاً پایدارتر از Trial است، اما سقف و مدل‌ها ممکن است تغییر کنند.

### مدل رایگان

فقط مدل‌های مشخصی بدون هزینه‌اند. سایر مدل‌ها Billing می‌خواهند. Routing یا نام مدل باید روشن باشد.

### Credit بدون کارت

مقدار کوچکی اعتبار دوره‌ای در اختیار حساب قرار می‌گیرد. پس از پایان Credit درخواست متوقف می‌شود یا نیاز به پرداخت دارد.

### Endpoint ناشناس

ممکن است بدون حساب و با کلید Placeholder کار کند. این گزینه برای آزمایش سریع مناسب است، اما Privacy، Abuse protection، پایداری و ظرفیت باید با دقت بیشتری بررسی شوند.

### Community Gateway

ثبت‌نام ممکن است خارج از Dashboard معمول و از طریق Community انجام شود. شرایط روزانه، صف مشترک و نگهداری Metadata را بررسی کنید.

## چک‌لیست بررسی واقعی

قبل از معرفی یک Provider به‌عنوان بدون کارت، این مراحل را ثبت کنید:

1. صفحه رسمی Pricing یا Free Tier را بخوانید.
2. راهنمای Signup را بررسی کنید.
3. حساب جدید با روش مجاز بسازید.
4. ببینید آیا Billing page اجباری است.
5. API Key ایجاد کنید.
6. مدل رایگان را مشخص کنید.
7. درخواست واقعی اجرا کنید.
8. سقف و خطاها را ثبت کنید.
9. تاریخ و Region را بنویسید.
10. اطلاعات حساس را حذف کنید.

اگر منبع رسمی صریح نیست و تست کامل انجام نشده، مقدار باید `unknown` بماند.

## نمونه استفاده امن

کلید را هرگز داخل سورس عمومی قرار ندهید. حتی اگر سرویس رایگان است، کلید می‌تواند برای مصرف سهمیه یا سوءاستفاده استفاده شود.

```python
import os
from openai import OpenAI

client = OpenAI(
    api_key=os.environ["FREE_LLM_API_KEY"],
    base_url=os.environ["FREE_LLM_BASE_URL"],
    timeout=25,
)

response = client.chat.completions.create(
    model=os.environ["FREE_LLM_MODEL"],
    messages=[
        {"role": "system", "content": "در سه جمله پاسخ بده."},
        {"role": "user", "content": "چرا API Key را نباید در Frontend قرار داد؟"},
    ],
)

print(response.choices[0].message.content)
```

فایل `.env` را در `.gitignore` قرار دهید. برای GitHub Actions از Repository یا Environment Secret استفاده کنید. Logها نباید Header احراز هویت یا Response کامل حساس را چاپ کنند.

## معیار سهمیه

بدون کارت‌بودن ارزشمند است، اما اگر سهمیه با کاربرد هماهنگ نباشد انتخاب مناسبی نیست. موارد زیر را مقایسه کنید:

- RPM؛
- RPD؛
- TPM؛
- سقف خروجی؛
- Context؛
- Concurrent request؛
- Reset period؛
- مدل‌های مجاز؛
- صف مشترک؛
- سیاست هنگام پایان سهمیه.

برای یک ربات عمومی، حتی Free Tier بزرگ نیز ممکن است کافی نباشد. برای نمونه اولیه، سهمیه کوچک با مستندات خوب می‌تواند بهتر از ظرفیت بالا و قرارداد مبهم باشد.

## کیفیت مدل را جدا بسنجید

جایگزین‌های GPT ممکن است از مدل‌های مختلف استفاده کنند. کیفیت را با Prompt واقعی بسنجید:

- پاسخ فارسی؛
- تولید کد؛
- خلاصه‌سازی؛
- استخراج JSON؛
- استدلال؛
- Tool Calling؛
- تبعیت از Context؛
- ثبات پاسخ.

از نام تجاری یا تعداد پارامتر نتیجه قطعی نگیرید. مدل کوچک‌تر ممکن است برای طبقه‌بندی سریع مناسب‌تر باشد و مدل بزرگ‌تر سهمیه را سریع مصرف کند.

## دسترسی ایران و کارت بانکی دو مسئله جدا هستند

یک Provider ممکن است کارت نخواهد ولی ایران را رسماً پشتیبانی نکند. یا Signup باز باشد اما Inference با IP ایران رد شود. بررسی باید این لایه‌ها را جدا نگه دارد:

- نیاز پرداخت؛
- نیاز هویت؛
- دسترسی Signup؛
- صدور Credential؛
- Reachability API؛
- Inference؛
- سیاست رسمی Region.

استفاده موفق از مسیر شبکه دیگر نباید به‌عنوان دسترسی مستقیم ثبت شود. این پروژه روش دورزدن محدودیت ارائه نمی‌کند.

## امنیت سرویس‌های ناشناس

اگر Endpoint بدون ثبت‌نام کار می‌کند، قبل از ارسال داده این سؤال‌ها را بپرسید:

- چه کسی سرویس را اداره می‌کند؟
- درخواست به کدام مدل Route می‌شود؟
- Prompt ذخیره می‌شود؟
- Metadata چه مدت نگه داشته می‌شود؟
- Abuse و Rate Limit چگونه مدیریت می‌شود؟
- آیا Terms و Privacy Policy وجود دارد؟
- آیا Endpoint ممکن است بدون اخطار تغییر کند؟
- آیا مدل برای داده محرمانه مناسب است؟

برای تمرین با متن عمومی ممکن است قابل قبول باشد، اما داده پزشکی، مالی، سازمانی یا Secret را ارسال نکنید.

## معماری مناسب برای Free API

Free Tier پویاست، پس برنامه باید قابل‌تعویض باشد:

1. Base URL، Key و Model را در Environment نگه دارید.
2. یک Adapter مشترک بسازید.
3. خطاها را Normalize کنید.
4. 429 را با Backoff مدیریت کنید.
5. Timeout داشته باشید.
6. پاسخ قابل Cache را ذخیره کنید.
7. Provider دوم برای مهاجرت آماده کنید.
8. مدل را با Alias داخلی صدا بزنید.
9. مصرف روزانه را اندازه بگیرید.
10. Terms و سهمیه را دوره‌ای بررسی کنید.

اگر Provider پرداخت را اجباری کرد، برنامه نباید بدون کنترل به مصرف پولی وارد شود. Budget cap و Fail-closed behavior داشته باشید.

## تفاوت Free Tier و Trial

Free Tier معمولاً دوره‌ای است، اما تضمین همیشگی ندارد. Trial محدود به زمان یا Credit است. برای پروژه آموزشی کوتاه هر دو مفیدند. برای محصول مداوم، Trial ریسک بیشتری دارد. در کاتالوگ نوع قرارداد باید صریح باشد.

## انتخاب برای سناریوهای مختلف

### یادگیری

سادگی Signup، مثال روشن و سهمیه کوچک کافی است. از داده عمومی استفاده کنید.

### پروژه دانشجویی

پایداری چند هفته، امکان دریافت Key و سقف روزانه مهم است. خروجی و تاریخ تست را مستند کنید.

### MVP

Adapter، Telemetry و Provider جایگزین از ابتدا اضافه کنید. از Trial به‌عنوان زیرساخت دائمی استفاده نکنید.

### محصول عمومی

Free Tier فقط بخشی از ظرفیت است. برنامه قیمت پس از رشد، SLA، Privacy و Support را بررسی کنید.

### پروژه سازمانی

نبود کارت مزیت اصلی نیست. قرارداد، امنیت، کنترل داده و پشتیبانی اهمیت بیشتری دارند.

## پرسش‌های متداول

### آیا API رایگان GPT واقعاً بدون کارت وجود دارد؟

برخی سرویس‌ها در شرایط جاری Payment Method نمی‌خواهند، اما قراردادها تغییر می‌کنند. تاریخ و منبع را بررسی کنید.

### آیا API ناشناس امن است؟

برای داده عمومی و تست محدود شاید قابل استفاده باشد، ولی برای داده حساس فقط با Privacy و قرارداد روشن تصمیم بگیرید.

### آیا بدون کارت می‌توان استفاده تجاری کرد؟

نیاز پرداخت و مجوز تجاری دو موضوع جدا هستند. Terms of Service را بخوانید.

### چرا مقدار نیاز کارت در کاتالوگ گاهی نامشخص است؟

چون منبع رسمی یا تست کافی وجود ندارد. مقدار `unknown` از ادعای بدون مدرک دقیق‌تر است.

### چگونه تغییر شرایط را مدیریت کنیم؟

Adapter، Provider جایگزین، Budget control و بررسی دوره‌ای داشته باشید. روی یک مدل موقت قفل نشوید.
