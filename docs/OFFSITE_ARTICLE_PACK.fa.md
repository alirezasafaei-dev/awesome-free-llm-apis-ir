# بسته مستقل مقالات انتشار خارجی

آخرین بازبینی: ۲۰۲۶-۰۷-۱۷

این سند یک بسته اجرایی مستقل برای انتشار مقاله در پلتفرم‌های فارسی است. این بسته عمداً هیچ تغییری در `ROADMAP`، اولویت‌های محصول، Issueهای P0/P1/P2 یا معیارهای بستن Issue #44 ایجاد نمی‌کند. ادغام این Pull Request فقط متن‌های آماده را نگهداری می‌کند؛ انتشار در حساب‌های بیرونی باید توسط مالک حساب انجام شود.

## هدف

- جذب کاربرانی که در فارسی درباره API رایگان LLM، API هوش مصنوعی، OpenAI-compatible، ساخت چت‌بات، Embedding، خطای 429 و دسترسی از ایران جست‌وجو می‌کنند.
- ساخت Mention و Backlink طبیعی از مقاله‌ای که به یک مسئله واقعی پاسخ می‌دهد.
- جلوگیری از انتشار ده نسخه تکراری یک متن و کاهش ریسک محتوای کم‌ارزش.
- هدایت کاربر به صفحه مرتبط در `https://llm.persiantoolbox.ir/` و سپس Catalog یا GitHub.

## وضعیت داده هنگام نگارش

در زمان تهیه این بسته، پروژه ۲۲ Provider را در Catalog اصلی نگهداری می‌کند. اطلاعاتی مانند نوع Free Tier، نیاز به کارت، RPM/RPD/TPM، سازگاری با OpenAI و وضعیت دسترسی ایران ممکن است تغییر کند؛ بنابراین عدد یا ادعای مربوط به Provider باید پیش از هر انتشار دوباره با سایت زنده کنترل شود.

## قواعد انتشار

1. هر مقاله فقط در پلتفرم تعیین‌شده منتشر شود؛ همان متن در چند دامنه کپی نشود.
2. نام پروژه و لینک مرجع طبیعی و محدود باشد؛ مقاله نباید شبیه تبلیغ یا Link farm شود.
3. عنوان، مقدمه و مثال‌ها متناسب با مخاطب همان سایت نگه داشته شوند.
4. API Key، Token، Cookie، IP کامل یا اطلاعات حساب در تصویر و متن منتشر نشود.
5. ادعای «قابل استفاده از ایران» فقط با تاریخ و عبارت دقیق پروژه مطرح شود.
6. قبل از انتشار، صفحه مقصد و پارامتر UTM باز و بررسی شود.
7. پس از انتشار، URL عمومی و تاریخ انتشار در Launch/Growth Log ثبت شود؛ بدون اضافه‌کردن آن به Roadmap.

# ۱. اُمیگو

## عنوان پیشنهادی

چطور یک API رایگان هوش مصنوعی انتخاب کنیم که واقعاً از ایران قابل استفاده باشد؟

## توضیح کوتاه

رایگان‌بودن یک API به معنی دائمی‌بودن سهمیه، نخواستن کارت بانکی یا کارکردن از ایران نیست. این راهنما یک چک‌لیست عملی برای انتخاب و تست APIهای LLM ارائه می‌کند.

## برچسب‌ها

هوش مصنوعی، API، برنامه‌نویسی، LLM، ابزار رایگان، توسعه نرم‌افزار

## لینک مقصد

`https://llm.persiantoolbox.ir/guides/practical-free-llm-api-iran/?utm_source=omigo&utm_medium=article&utm_campaign=offsite_articles&utm_content=practical_api_selection`

## متن آماده انتشار

عبارت «API رایگان هوش مصنوعی» در نگاه اول ساده به نظر می‌رسد: یک سرویس پیدا می‌کنیم، کلید می‌گیریم و درخواست می‌فرستیم. مشکل زمانی شروع می‌شود که می‌بینیم رایگان‌بودن در سرویس‌های مختلف معنای یکسانی ندارد.

ممکن است یک Provider فقط چند دلار اعتبار اولیه بدهد، سرویس دیگری دوره آزمایشی کوتاه داشته باشد و سرویسی دیگر مدل‌های مشخصی را با محدودیت روزانه رایگان ارائه کند. حتی اگر سهمیه مناسب باشد، ممکن است ثبت‌نام از ایران کامل نشود یا درخواست واقعی مدل با محدودیت منطقه‌ای روبه‌رو شود.

برای انتخاب درست، بهتر است به‌جای تکیه بر فهرست‌های قدیمی یا توصیه‌های بدون تاریخ، شش مرحله را بررسی کنیم.

### ۱. نوع رایگان‌بودن را مشخص کنید

چهار عبارت معمولاً با یکدیگر اشتباه گرفته می‌شوند:

- **Free Tier دائمی:** سهمیه‌ای که به‌صورت مستمر ارائه می‌شود، هرچند سیاست آن ممکن است در آینده تغییر کند.
- **Free Models:** فقط بعضی مدل‌ها یا Endpointها رایگان‌اند.
- **Credit:** اعتبار مالی محدود که پس از مصرف تمام می‌شود.
- **Trial:** دسترسی آزمایشی زمان‌دار یا حساب‌محور.

اگر پروژه شما قرار است بیشتر از یک آزمایش کوتاه عمر کند، اتکا به Trial تصمیم مطمئنی نیست.

### ۲. نیاز به کارت بانکی را جدا بررسی کنید

گاهی صفحه قیمت‌گذاری عبارت Free را نشان می‌دهد، اما برای فعال‌سازی حساب یا عبور از محدودیت اولیه به Payment Method نیاز است. برای کاربران ایرانی این موضوع می‌تواند کل مسیر را متوقف کند. بنابراین «سهمیه رایگان» و «بدون کارت بانکی» دو ویژگی جدا هستند.

### ۳. ثبت‌نام، ساخت کلید و اجرای مدل را یکی ندانید

بازشدن صفحه اصلی سرویس کافی نیست. یک تست واقعی حداقل چهار مرحله دارد:

1. ساخت حساب؛
2. ساخت API Key؛
3. انتخاب Model ID معتبر؛
4. دریافت پاسخ موفق از Endpoint مدل.

ممکن است سه مرحله اول موفق باشند اما مدل برای منطقه، حساب یا پلن فعال نباشد.

### ۴. محدودیت مصرف را با نیاز پروژه تطبیق دهید

اعدادی مانند RPM، RPD، TPM و محدودیت هم‌زمانی مستقیماً روی تجربه کاربر اثر می‌گذارند:

- RPM: تعداد درخواست در دقیقه؛
- RPD: تعداد درخواست در روز؛
- TPM: تعداد توکن در دقیقه؛
- Context limit: حداکثر ورودی قابل پردازش؛
- Concurrent requests: تعداد درخواست هم‌زمان.

برای یک چت‌بات شخصی شاید سهمیه کم کافی باشد، اما برای کلاس آموزشی، افزونه سایت یا MVP چندکاربره ممکن است همان سهمیه در چند دقیقه تمام شود.

### ۵. سازگاری با OpenAI API را بررسی کنید

اگر سرویس OpenAI-compatible باشد، معمولاً می‌توان با تغییر `base_url`، `api_key` و `model` از SDKهای رایج استفاده کرد. این ویژگی تعویض Provider را ساده‌تر می‌کند، اما به معنی یکسان‌بودن همه قابلیت‌ها نیست. Tool calling، JSON output، Streaming و نام مدل‌ها باید جداگانه تست شوند.

### ۶. وضعیت ایران را تاریخ‌دار ببینید

دسترسی منطقه‌ای ثابت نیست. نتیجه ممکن است میان اینترنت ثابت، موبایل، ISPهای مختلف و زمان‌های متفاوت فرق کند. عبارت «برای من کار کرد» بدون تاریخ، نوع شبکه و درخواست واقعی، مدرک کافی نیست.

پروژه متن‌باز **Awesome Free LLM APIs IR** این اطلاعات را در یک Catalog فارسی و ماشین‌خوان نگهداری می‌کند و Free Tier، نیاز به پرداخت، محدودیت مصرف، سازگاری OpenAI و وضعیت مستقیم یا مسدود از ایران را از یکدیگر جدا نشان می‌دهد.

پیش از انتخاب نهایی، راهنمای مرحله‌به‌مرحله و فهرست زنده را از لینک زیر ببینید:

[راهنمای عملی انتخاب API رایگان LLM از ایران](https://llm.persiantoolbox.ir/guides/practical-free-llm-api-iran/?utm_source=omigo&utm_medium=article&utm_campaign=offsite_articles&utm_content=practical_api_selection)

کد منبع و داده‌های پروژه نیز در GitHub عمومی است:

`https://github.com/alirezasafaei-dev/awesome-free-llm-apis-ir`

نکته پایانی: برای پروژه واقعی، Provider را فقط به‌دلیل نام مدل انتخاب نکنید. سهمیه، دسترسی، روش ثبت‌نام، امکان تعویض سرویس و رفتار خطاها به همان اندازه مهم‌اند.

# ۲. بلاگیکس

## عنوان پیشنهادی

API سازگار با OpenAI بدون کارت بانکی؛ از ادعای رایگان تا تست واقعی

## توضیح کوتاه

OpenAI-compatible بودن چه معنایی دارد و چگونه می‌توان یک API رایگان را بدون قفل‌شدن به یک Provider وارد پروژه کرد؟

## برچسب‌ها

OpenAI API، استارتاپ، Python، Node.js، LLM، API رایگان

## لینک مقصد

`https://llm.persiantoolbox.ir/guides/openai-compatible-api-without-card/?utm_source=blogix&utm_medium=article&utm_campaign=offsite_articles&utm_content=openai_compatible_no_card`

## متن آماده انتشار

بسیاری از پروژه‌های هوش مصنوعی با یک نمونه کد ساده شروع می‌شوند، اما بعد از چند روز با سه مانع روبه‌رو می‌شوند: تمام‌شدن اعتبار آزمایشی، نیاز به کارت بانکی و وابستگی شدید کد به یک Provider.

یک راه کاهش این ریسک، استفاده از APIهای **OpenAI-compatible** است. منظور این نیست که سرویس متعلق به OpenAI است؛ بلکه ساختار Endpoint و درخواست آن تا حدی با قرارداد رایج OpenAI سازگار است. در نتیجه می‌توان همان SDK را نگه داشت و با تغییر چند تنظیم، Provider دیگری را آزمایش کرد.

### OpenAI-compatible دقیقاً چه چیزی را ساده می‌کند؟

در حالت معمول سه مقدار تغییر می‌کنند:

```text
API Key
Base URL
Model ID
```

برای مثال در Python می‌توان تنظیمات را از Environment Variable خواند:

```python
import os
from openai import OpenAI

client = OpenAI(
    api_key=os.environ["LLM_API_KEY"],
    base_url=os.environ["LLM_BASE_URL"],
)

response = client.chat.completions.create(
    model=os.environ["LLM_MODEL"],
    messages=[
        {"role": "user", "content": "این متن را در سه نکته خلاصه کن."}
    ],
)

print(response.choices[0].message.content)
```

این ساختار یک مزیت مهم دارد: کلید و آدرس سرویس داخل کد ثابت نمی‌شوند و تعویض Provider به بازنویسی کامل برنامه نیاز ندارد.

### سازگاری اسمی کافی نیست

دو سرویس ممکن است هر دو Endpoint سازگار با OpenAI داشته باشند، اما رفتار یکسانی نشان ندهند. پیش از انتخاب باید این موارد تست شوند:

- Streaming؛
- Tool calling؛
- Structured output یا JSON mode؛
- Embedding؛
- طول Context؛
- نام و نسخه مدل؛
- شکل خطا و Rate limit.

بنابراین بهتر است یک تست Contract کوچک در پروژه داشته باشید که پس از تغییر Provider، پاسخ ساده، Streaming و خطاها را کنترل کند.

### «بدون کارت» را جدا از «رایگان» بررسی کنید

در زمان نگارش این مقاله، Catalog پروژه نمونه‌هایی از Gatewayها و Providerهای رسمی را ثبت کرده است که Free Tier یا مدل رایگان دارند و برای فعال‌سازی اولیه Payment Method نمی‌خواهند. بااین‌حال سهمیه و شرایط حساب‌ها ثابت نیست و باید تاریخ آخرین بررسی را دید.

برای نمونه، داده تاریخ‌دار پروژه در میانه ژوئیه ۲۰۲۶ نشان می‌داد:

- Cloudflare Workers AI سهمیه روزانه مبتنی بر Neuron دارد و در رکورد پروژه بدون Payment Method ثبت شده است؛
- GitHub Models برای Preview رایگان محدودیت RPM و RPD دارد و با Token حساب GitHub کار می‌کند؛
- Mistral AI حالت Free بدون کارت ارائه می‌دهد، اما عدد محدودیت می‌تواند بر اساس حساب و مدل در کنسول نمایش داده شود.

این مثال‌ها توصیه قطعی یا تضمین همیشگی نیستند. ارزش Catalog در این است که نوع سهمیه، منبع رسمی، تاریخ بررسی و وضعیت ایران را کنار هم نشان می‌دهد.

### معماری پیشنهادی برای جلوگیری از Vendor lock-in

بهتر است لایه فراخوانی مدل را از منطق اصلی برنامه جدا کنید:

```text
UI / API Route
      ↓
LLM Service Interface
      ↓
Provider Adapter
      ↓
OpenAI-compatible Endpoint
```

در این معماری، کد اصلی فقط یک تابع مثل `generate_text()` یا `chat()` را می‌شناسد. Adapter مسئول Base URL، Model ID، Timeout و تبدیل خطاها است.

### چک‌لیست قبل از استفاده

1. نوع Free Tier را بخوانید؛
2. نیاز به کارت یا پرداخت را کنترل کنید؛
3. یک API Key جدید و محدود بسازید؛
4. درخواست واقعی مدل را اجرا کنید؛
5. خطاهای 401، 403 و 429 را تست کنید؛
6. Secret را فقط در Backend نگه دارید؛
7. Provider دوم را به‌عنوان مسیر جایگزین آماده کنید.

فهرست به‌روز سرویس‌های OpenAI-compatible و وضعیت بدون کارت را می‌توانید در راهنمای زیر مقایسه کنید:

# ۳. بلاگ بیان (Blog.ir)

## عنوان پیشنهادی

چرا بیشتر فهرست‌های API رایگان هوش مصنوعی خیلی زود قدیمی می‌شوند؟

## توضیح کوتاه

یک فهرست قابل اعتماد باید میان Provider رسمی، Gateway، Trial، سهمیه دائمی و نتیجه واقعی دسترسی از ایران تفاوت بگذارد.

## برچسب‌ها

داده باز، هوش مصنوعی، API، متن‌باز، GitHub، توسعه‌دهندگان ایرانی

## لینک مقصد

`https://llm.persiantoolbox.ir/?utm_source=blog_ir&utm_medium=article&utm_campaign=offsite_articles&utm_content=why_free_api_lists_expire`

## متن آماده انتشار

فهرست‌های «بهترین API رایگان هوش مصنوعی» معمولاً در روز انتشار مفیدند، اما چند ماه بعد بخشی از اطلاعات آن‌ها دیگر درست نیست. یک سرویس Trial خود را حذف می‌کند، Provider دیگری محدودیت درخواست را تغییر می‌دهد، نام مدل منسوخ می‌شود یا ثبت‌نام از یک منطقه مسدود می‌گردد.

مشکل فقط تغییر بازار نیست؛ شیوه ثبت داده نیز اهمیت دارد. وقتی همه انواع رایگان‌بودن در یک ستون با علامت تیک نمایش داده شوند، کاربر نمی‌فهمد با سهمیه دائمی، اعتبار اولیه یا یک دوره آزمایشی روبه‌رو است.

### یک رکورد قابل اعتماد چه اجزایی دارد؟

برای هر سرویس حداقل باید این موارد جدا ثبت شوند:

- نوع سرویس: Provider رسمی، Gateway رسمی یا Gateway اجتماعی؛
- نوع دسترسی رایگان: دائمی، مدل رایگان، Credit یا Trial؛
- نیاز به کارت بانکی یا پرداخت اولیه؛
- محدودیت‌های RPM، RPD، TPM و توکن؛
- Base URL و شیوه Authentication؛
- سازگاری با OpenAI API؛
- قابلیت‌ها مانند Chat، Reasoning، Embedding و Tool calling؛
- تاریخ آخرین بررسی؛
- منبع رسمی هر ادعا.

بدون این تفکیک، عبارت «رایگان» ارزش تصمیم‌گیری چندانی ندارد.

### وضعیت ایران یک فیلد بله/خیر نیست

برای کاربران ایرانی، بازشدن سایت یا حتی پاسخ‌دادن یک Endpoint عمومی به معنی کارکرد مدل نیست. مسیر واقعی شامل ثبت‌نام، ساخت کلید و اجرای درخواست مدل است.

بهتر است چند وضعیت از هم جدا شوند:

- درخواست مستقیم از شبکه ایران موفق بوده است؛
- دسترسی مستقیم شکست خورده اما مسیر VPN موفق بوده است؛
- سیاست رسمی منطقه را پشتیبانی نمی‌کند؛
- ثبت‌نام مسدود است؛
- نتیجه میان شبکه‌ها ناپایدار است؛
- شواهد تازه کافی وجود ندارد.

در این روش، «نامشخص» به معنی کارکردن سرویس نیست؛ فقط یعنی داده معتبر کافی در دسترس نیست.

### چرا تاریخ و Evidence مهم‌اند؟

یک گزارش خوب باید پاسخ دهد:

- چه زمانی تست انجام شده؟
- درخواست واقعی کدام مدل بوده؟
- نتیجه HTTP چه بوده؟
- مسیر مستقیم بوده یا VPN؟
- نوع شبکه ثابت بوده یا موبایل؟
- منبع رسمی سهمیه کجاست؟

در عین حال نباید API Key، IP کامل، Cookie یا اطلاعات هویتی منتشر شود. هدف، قابل بررسی‌کردن ادعاست نه افشای حساب کاربر.

### داده ماشین‌خوان چه کمکی می‌کند؟

# ۴. بلاگفا

## عنوان پیشنهادی

فرق Free Tier، Trial و Credit در APIهای هوش مصنوعی چیست؟

## توضیح کوتاه

سه اصطلاحی که هنگام انتخاب API رایگان زیاد دیده می‌شوند، اما هزینه و دوام کاملاً متفاوتی دارند.

## متن آماده انتشار

هنگام جست‌وجوی API رایگان هوش مصنوعی معمولاً با سه عبارت روبه‌رو می‌شویم: **Free Tier**، **Trial** و **Credit**. هر سه ممکن است امکان شروع بدون پرداخت فوری را بدهند، اما برای برنامه‌ریزی یک پروژه معنای یکسانی ندارند.

### Free Tier چیست؟

Free Tier سهمیه‌ای است که سرویس برای استفاده مداوم در نظر می‌گیرد. این سهمیه ممکن است روزانه یا ماهانه بازنشانی شود و معمولاً محدودیت‌هایی مانند تعداد درخواست، توکن یا مدل دارد.

مثلاً ممکن است حساب رایگان اجازه دهد روزانه تعداد مشخصی درخواست بفرستید یا فقط از چند مدل کوچک‌تر استفاده کنید. Free Tier دائمی به معنی تضمین همیشگی نیست؛ Provider می‌تواند سیاست قیمت‌گذاری را تغییر دهد. بااین‌حال برای پروژه‌های آموزشی و نمونه اولیه از Trial قابل اتکاتر است.

### Trial چیست؟

Trial یک دوره آزمایشی است که معمولاً با یکی از این دو شرط پایان می‌یابد:

- گذشتن زمان مشخص، مانند چند روز یا یک ماه؛
- رسیدن مصرف به سقف تعیین‌شده.

Trial برای ارزیابی کیفیت مدل مناسب است، اما اگر برنامه شما باید ماه‌ها کار کند، نباید معماری و هزینه را بر ادامه Trial بنا کنید.

### Credit چیست؟

Credit اعتبار مالی است. برای مثال ممکن است حساب جدید مقدار محدودی اعتبار دریافت کند و مصرف هر مدل از آن کم شود. تفاوت Credit با سهمیه دائمی این است که پس از تمام‌شدن اعتبار، ادامه استفاده معمولاً نیازمند پرداخت است.

گاهی Credit ماهانه دوباره شارژ می‌شود و گاهی فقط یک بار ارائه می‌گردد. همچنین ممکن است فعال‌سازی آن به ثبت Payment Method وابسته باشد.

### Free Models یعنی چه؟

بعضی Gatewayها همه سرویس را رایگان نمی‌کنند؛ فقط مدل‌های خاصی با پسوند یا برچسب Free در دسترس‌اند. محدودیت این مدل‌ها می‌تواند با مدل‌های پولی متفاوت باشد و صف، Rate limit یا ظرفیت متغیر داشته باشند.

### چرا نیاز به کارت بانکی باید جدا بررسی شود؟

ممکن است سرویس Free Tier داشته باشد اما برای ساخت حساب، فعال‌سازی API یا جلوگیری از سوءاستفاده کارت بانکی بخواهد. برای کاربران ایرانی، همین شرط می‌تواند استفاده عملی را ناممکن کند.

پس هنگام مقایسه، این دو سؤال را جدا بپرسید:

1. آیا مصرف رایگان وجود دارد؟
2. آیا برای فعال‌سازی آن روش پرداخت لازم است؟

### محدودیت‌ها را چگونه بخوانیم؟

چند مخفف رایج:

- **RPM:** درخواست در دقیقه؛
- **RPD:** درخواست در روز؛
- **TPM:** توکن در دقیقه؛
- **TPD:** توکن در روز.

اگر RPD کم باشد، حتی RPM بالا هم پروژه چندکاربره را نجات نمی‌دهد. اگر TPM پایین باشد، درخواست‌های طولانی سریعاً به خطای Rate limit می‌رسند.

### برای هر نوع پروژه کدام بهتر است؟

- برای یادگیری و تمرین: Trial یا Credit هم می‌تواند کافی باشد.
- برای پایان‌نامه و پروژه دانشجویی چندماهه: Free Tier دائمی یا مدل‌های رایگان پایدار مناسب‌ترند.
- برای MVP: Free Tier برای شروع خوب است، اما باید هزینه پلن بعدی و Provider جایگزین را بدانید.
- برای سرویس تجاری: هیچ سهمیه رایگانی را بدون مانیتورینگ، بودجه و Fallback مبنای تعهد به مشتری قرار ندهید.

## متن آماده انتشار

برای آزمایش یک مدل زبانی لازم نیست کد را از ابتدا به SDK اختصاصی هر Provider وابسته کنیم. بسیاری از سرویس‌ها Endpoint سازگار با OpenAI دارند و با تغییر Base URL، API Key و Model ID قابل استفاده‌اند.

در این آموزش یک نمونه ساده می‌سازیم که Secret را از متغیر محیطی می‌خواند و بعداً می‌توان Provider آن را بدون تغییر منطق برنامه عوض کرد.

### نصب کتابخانه

```bash
python -m pip install --upgrade openai
```

### تعریف متغیرهای محیطی

در Linux یا macOS:

```bash
export LLM_API_KEY="your-key"
export LLM_BASE_URL="https://provider.example/v1"
export LLM_MODEL="provider-model-id"
```

در PowerShell:

```powershell
$env:LLM_API_KEY="your-key"
$env:LLM_BASE_URL="https://provider.example/v1"
$env:LLM_MODEL="provider-model-id"
```

کلید واقعی را داخل فایل Python، اسکرین‌شات یا Repository قرار ندهید. اگر از فایل `.env` استفاده می‌کنید، آن را در `.gitignore` قرار دهید.

### کد پایه

```python
import os
import sys
from openai import OpenAI, APIConnectionError, AuthenticationError, RateLimitError


def required_env(name: str) -> str:
    value = os.getenv(name)
    if not value:
        raise RuntimeError(f"Missing environment variable: {name}")
    return value


client = OpenAI(
    api_key=required_env("LLM_API_KEY"),
    base_url=required_env("LLM_BASE_URL"),
    timeout=30.0,
    max_retries=2,
)

try:
    response = client.chat.completions.create(
        model=required_env("LLM_MODEL"),
        messages=[
            {
                "role": "system",
                "content": "پاسخ را دقیق، کوتاه و به زبان فارسی بنویس.",
            },
            {
                "role": "user",
                "content": "سه کاربرد عملی مدل‌های زبانی برای دانشجوها را بگو.",
            },
        ],
        temperature=0.3,
    )

    print(response.choices[0].message.content)
except AuthenticationError:
    print("خطای احراز هویت: API Key یا مجوز مدل را بررسی کنید.", file=sys.stderr)
except RateLimitError:
    print("محدودیت مصرف: کمی صبر کنید یا سهمیه حساب را بررسی کنید.", file=sys.stderr)
except APIConnectionError:
    print("خطای شبکه یا Base URL: اتصال و آدرس Endpoint را بررسی کنید.", file=sys.stderr)
```

### چرا تنظیمات را بیرون کد نگه می‌داریم؟

این کار سه مزیت دارد:

1. Secret وارد Git نمی‌شود؛
2. محیط توسعه و Production تنظیمات جدا دارند؛
3. تعویض Provider بدون تغییر فایل اصلی انجام می‌شود.

### خطاهای رایج

**401 Unauthorized:** کلید اشتباه، منقضی یا فاقد مجوز است.

**403 Forbidden:** حساب یا منطقه اجازه استفاده از مدل را ندارد.

**404 / Model not found:** Model ID یا Base URL درست نیست.

**429 Too Many Requests:** محدودیت RPM، RPD یا TPM رد شده است.

**Timeout:** شبکه، ظرفیت Provider یا طول درخواست مشکل دارد.

### آیا هر Provider سازگار دقیقاً یکسان است؟

خیر. پارامترهایی مانند `temperature`، Tool calling، JSON mode و Streaming ممکن است در همه سرویس‌ها یکسان نباشند. ابتدا یک درخواست کوچک بفرستید و سپس قابلیت‌های پیشرفته را جدا تست کنید.

### انتخاب Provider رایگان

قبل از ساخت کلید، این موارد را بررسی کنید:

- سهمیه دائمی است یا Trial؟
- Payment Method لازم دارد؟
- از ایران ثبت‌نام و درخواست واقعی ممکن است؟
- مدل موردنظر روی حساب رایگان فعال است؟
- محدودیت Context و Rate limit برای پروژه کافی است؟

# ۶. فارسی‌بلاگ

## عنوان پیشنهادی

چطور یک چت‌بات فارسی بسازیم که به یک سرویس هوش مصنوعی وابسته نباشد؟

## توضیح کوتاه

## متن آماده انتشار

ساخت نسخه اولیه یک چت‌بات با مدل زبانی آسان است، اما پایدارکردن آن به چند تصمیم معماری نیاز دارد. اگر همه چیز مستقیماً به SDK و نام مدل یک Provider وصل شود، تغییر قیمت، محدودیت منطقه‌ای یا حذف مدل می‌تواند کل برنامه را متوقف کند.

راه بهتر این است که منطق چت‌بات را از Provider جدا کنیم.

### چهار بخش اصلی چت‌بات

```text
رابط کاربر
مدیریت گفتگو
سرویس LLM
Adapter مربوط به Provider
```

رابط کاربر می‌تواند ترمینال، سایت یا پیام‌رسان باشد. مدیریت گفتگو مسئول نگهداری چند پیام آخر است. سرویس LLM یک قرارداد ثابت ارائه می‌دهد و Adapter جزئیات Base URL، کلید و Model ID را می‌داند.

### قرارداد ساده سرویس

```python
from typing import Protocol


class ChatProvider(Protocol):
    def reply(self, messages: list[dict[str, str]]) -> str:
        ...
```

منطق برنامه فقط متد `reply` را صدا می‌زند. در نتیجه می‌توان Adapter مربوط به Provider اول را با Adapter دیگری جایگزین کرد.

### مدیریت تاریخچه

ارسال تمام پیام‌های گذشته باعث افزایش توکن و هزینه می‌شود. برای نسخه ساده می‌توان فقط چند پیام آخر را نگه داشت:

```python
MAX_MESSAGES = 12
history = history[-MAX_MESSAGES:]
```

برای گفتگوهای طولانی‌تر، خلاصه‌سازی دوره‌ای یا ذخیره اطلاعات مهم در پایگاه داده بهتر است.

### System Prompt فارسی

System Prompt باید کوتاه و مشخص باشد. برای نمونه:

```text
تو یک دستیار فارسی هستی. پاسخ‌ها را دقیق و روشن بنویس.
هنگام نبود اطلاعات کافی، حدس قطعی نزن.
اطلاعات محرمانه یا کلید API درخواست نکن.
```

دستورهای بسیار طولانی هم توکن بیشتری مصرف می‌کنند و هم احتمال تعارض را بالا می‌برند.

### مدیریت خطا برای تجربه کاربری بهتر

نباید متن خام خطای Provider مستقیماً به کاربر نشان داده شود. بهتر است خطاها به چند گروه تبدیل شوند:

- احراز هویت: تنظیمات سرویس مشکل دارد؛
- محدودیت مصرف: کمی بعد دوباره تلاش شود؛
- مدل نامعتبر: Model ID نیاز به بازبینی دارد؛
- شبکه: اتصال موقتاً برقرار نیست؛
- خطای داخلی Provider: مسیر جایگزین فعال شود.

Retry باید محدود و همراه Backoff باشد. تکرار نامحدود درخواست 429 فقط سهمیه را بدتر می‌کند.

### امنیت API Key

کلید باید فقط در Backend نگه‌داری شود. قراردادن کلید در JavaScript مرورگر، اپ موبایل بدون Backend، Repository عمومی یا تصویر آموزشی ناامن است.

برای محیط‌های مختلف از Secret Manager یا Environment Variable استفاده کنید و دسترسی کلید را حداقل نگه دارید.

### Provider رایگان را چگونه انتخاب کنیم؟

برای یک چت‌بات فارسی این معیارها مهم‌اند:

- کیفیت پاسخ فارسی؛
- Context کافی؛
- Rate limit مناسب؛
- Streaming در صورت نیاز؛
- دسترسی عملی از ایران؛
- سازگاری با SDK موجود؛
- امکان استفاده از مدل جایگزین.

کیفیت فارسی را با چند Prompt واقعی پروژه بسنجید. نام یا اندازه مدل به‌تنهایی تضمین نمی‌کند که لحن، دستورپذیری یا نگارش فارسی مناسب باشد.

### Fallback ساده

برای شروع می‌توان دو Provider تعریف کرد. فقط خطاهای موقت مانند Timeout یا خطای ظرفیت باید مسیر دوم را فعال کنند. خطای Authentication نباید خودکار به Provider بعدی منتقل شود، چون ممکن است نشانه تنظیمات اشتباه یا مشکل امنیتی باشد.

### سناریوی استفاده را دقیق تعریف کنید

### معیارهای اصلی انتخاب

#### ۱. کیفیت Reasoning

برای رفع باگ و Refactor، مدل باید بتواند محدودیت‌ها را دنبال کند و میان چند بخش کد ارتباط برقرار نماید. این ویژگی را با Benchmark عمومی به‌تنهایی نسنجید؛ نمونه‌های واقعی زبان و Framework خودتان را امتحان کنید.

#### ۲. طول Context

اگر قرار است چند فایل یا Log طولانی ارسال شود، Context محدود سریعاً مانع می‌شود. در عین حال ارسال کل Repository هم روش خوبی نیست. فایل‌های مرتبط، خطا و تست شکست‌خورده را انتخاب کنید.

#### ۳. خروجی ساخت‌یافته

برای Agentها یا ابزارهای خودکار، JSON معتبر و Tool calling مهم‌تر از نثر زیباست. Endpoint باید این قابلیت را واقعاً پشتیبانی کند، نه اینکه فقط از نظر URL سازگار باشد.

#### ۴. Rate limit

یک Agent ممکن است برای یک کار چندین درخواست پشت سر هم بفرستد. RPD پایین یا محدودیت Concurrent request می‌تواند روند را متوقف کند. RPM، RPD و TPM را کنار هم ببینید.

#### ۵. سرعت و پایداری

برای تکمیل تعاملی کد، Latency مهم است. برای تحلیل دسته‌ای، پایداری و سهمیه اهمیت بیشتری دارد. بهتر است زمان پاسخ و نرخ خطا را در چند ساعت مختلف ثبت کنید.

## متن آماده انتشار

در جست‌وجوی معمولی، سیستم به دنبال واژه‌های یکسان می‌گردد. اما کاربر ممکن است سؤال «چطور رمز حسابم را عوض کنم؟» را بنویسد، در حالی که متن راهنما با عنوان «بازیابی و تغییر گذرواژه» ذخیره شده است. جست‌وجوی معنایی تلاش می‌کند مفهوم دو متن را مقایسه کند، نه فقط کلمات مشترک را.

ابزار اصلی این کار **Embedding** است.

### Embedding چیست؟

مدل Embedding هر متن را به یک بردار عددی تبدیل می‌کند. متن‌هایی که از نظر معنا نزدیک‌اند، معمولاً بردارهای نزدیک‌تری دارند. سپس می‌توان با معیارهایی مانند Cosine similarity نزدیک‌ترین متن‌ها را پیدا کرد.

فرآیند ساده:

```text
متن‌ها → Embedding → ذخیره بردارها
سؤال کاربر → Embedding → جست‌وجوی نزدیک‌ترین بردارها
```

### کاربردها

- جست‌وجوی معنایی در اسناد؛
- پیشنهاد محتوای مشابه؛
- دسته‌بندی متن؛
- تشخیص شباهت؛
- بازیابی Context برای RAG؛
- خوشه‌بندی پرسش‌های کاربران.

### RAG چگونه از Embedding استفاده می‌کند؟

در معماری RAG ابتدا بخش‌های مرتبط از اسناد بازیابی می‌شوند و سپس همراه سؤال به مدل زبانی داده می‌شوند. این روش کمک می‌کند پاسخ بر اساس محتوای مشخص تولید شود.

```text
سؤال
 ↓
Embedding سؤال
 ↓
Vector Search
 ↓
بخش‌های مرتبط
 ↓
LLM
 ↓
پاسخ همراه Context
```

### نکات مهم برای زبان فارسی

کیفیت یک مدل Embedding در انگلیسی الزاماً به معنی عملکرد خوب در فارسی نیست. بهتر است مجموعه‌ای کوچک از پرسش‌ها و پاسخ‌های واقعی فارسی بسازید و معیارهایی مانند Recall@k را اندازه بگیرید.

موارد زیر روی نتیجه اثر دارند:

- نیم‌فاصله و شکل حروف؛
- «ی» و «ک» عربی یا فارسی؛
- علائم نگارشی؛
- طول Chunk؛
- متن محاوره‌ای یا رسمی؛
- چندزبانه‌بودن مدل.

پیش‌پردازش باید محدود و قابل تکرار باشد. حذف افراطی کلمات یا تغییر متن ممکن است معنی را خراب کند.

### Chunking

سند طولانی باید به بخش‌های کوچک تقسیم شود. Chunk خیلی کوتاه Context کافی ندارد و Chunk بسیار بلند موضوعات مختلف را مخلوط می‌کند. برای شروع می‌توان چند اندازه را روی داده واقعی مقایسه کرد و مقداری همپوشانی در نظر گرفت.

# ۹. تک‌بلاگ

## عنوان پیشنهادی

## لینک مقصد

## متن آماده انتشار

وقتی یک درخواست LLM شکست می‌خورد، تغییر تصادفی API Key، Model ID و VPN معمولاً تشخیص را سخت‌تر می‌کند. بهتر است خطا را مرحله‌به‌مرحله دسته‌بندی کنیم و هر بار فقط یک متغیر را تغییر دهیم.

### ابتدا یک درخواست حداقلی بسازید

برای عیب‌یابی، Prompt کوتاه، Streaming خاموش و بدون Tool calling استفاده کنید. همچنین Base URL، نام مدل و مسیر Endpoint را از مستندات همان Provider بردارید.

هرگز API Key را داخل Log عمومی چاپ نکنید.

### خطای 401 Unauthorized

معمولاً به Authentication مربوط است:

- کلید اشتباه یا ناقص؛
- فاصله یا Quote اضافی در Environment Variable؛
- کلید منقضی یا حذف‌شده؛
- Header نامعتبر؛
- استفاده از Token نوع دیگر.

راه‌حل:

1. کلید جدید محدود بسازید؛
2. مقدار Environment Variable را بدون نمایش کامل بررسی کنید؛
3. نمونه رسمی `curl` را اجرا کنید؛
4. سپس همان تنظیمات را وارد SDK کنید.

### خطای 403 Forbidden

در 403 هویت شناخته شده اما اجازه دسترسی داده نشده است. دلایل رایج:

- مدل برای حساب فعال نیست؛
- پلن یا Preview لازم فعال نشده؛
- منطقه پشتیبانی نمی‌شود؛
- Account policy یا محدودیت سازمانی وجود دارد؛
- Endpoint به Scope خاصی نیاز دارد.

در این حالت عوض‌کردن تصادفی Model ID کافی نیست. صفحه حساب، سیاست منطقه‌ای و مجوز Token را بررسی کنید.

### خطای 404 یا Model Not Found

این خطا معمولاً از یکی از موارد زیر می‌آید:

- Base URL اشتباه؛
- مسیر `/v1` جا افتاده یا دوبار تکرار شده؛
- نام مدل منسوخ شده؛
- مدل فقط در منطقه یا Endpoint دیگری وجود دارد؛
- Provider نام Alias متفاوتی استفاده می‌کند.

فهرست مدل‌های حساب را از API یا Console بگیرید و به مقاله‌های قدیمی اعتماد نکنید. مدل‌ها ممکن است بدون تغییر نام Provider حذف یا جایگزین شوند.

### خطای 429 Too Many Requests

429 فقط به معنی «درخواست زیاد در یک ثانیه» نیست. ممکن است یکی از محدودیت‌های زیر رد شده باشد:

- RPM؛
- RPD؛
- TPM؛
- TPD؛
- Concurrent requests؛
- اعتبار یا سهمیه حساب.

راه‌حل مناسب:

- Headerهای Rate limit را ثبت کنید؛
- Retry محدود با Exponential backoff داشته باشید؛
- به `Retry-After` احترام بگذارید؛
- درخواست‌های تکراری را Cache کنید؛
- Prompt و Context را کوتاه‌تر کنید؛
- صف و محدودیت هم‌زمانی اضافه کنید.

Retry فوری و نامحدود می‌تواند وضعیت را بدتر کند.

### Timeout و خطای شبکه

Timeout ممکن است از اینترنت کاربر، DNS، Proxy، ظرفیت Provider یا پاسخ طولانی ناشی شود. برای تشخیص:

1. Endpoint را با `curl` تست کنید؛
2. همان درخواست را روی یک شبکه دیگر اجرا کنید؛
3. Timeout منطقی تنظیم کنید؛
4. Prompt کوتاه و مدل سبک‌تر را امتحان کنید؛
5. زمان و نوع شبکه را ثبت کنید.

### وضعیت ایران را چگونه تفسیر کنیم؟

بازشدن Dashboard با موفقیت API برابر نیست. تست کامل شامل ساخت کلید و دریافت پاسخ مدل است. همچنین موفقیت با VPN به معنی دسترسی مستقیم نیست و شکست یک ISP را نباید بدون شواهد به همه شبکه‌های ایران تعمیم داد.

# ۱۰. رزبلاگ

## عنوان پیشنهادی

معماری چند Provider برای ابزار هوش مصنوعی؛ Fallback بدون آشفتگی و هزینه پنهان

## توضیح کوتاه

چگونه یک MVP هوش مصنوعی را طوری طراحی کنیم که با تغییر سهمیه، اختلال یا محدودیت منطقه‌ای یک Provider از کار نیفتد؟

## برچسب‌ها

معماری نرم‌افزار، MVP، هوش مصنوعی، API، Fallback، استارتاپ

## لینک مقصد

`https://llm.persiantoolbox.ir/?utm_source=rozblog&utm_medium=article&utm_campaign=offsite_articles&utm_content=multi_provider_fallback_architecture`

## متن آماده انتشار

اتصال مستقیم تمام بخش‌های برنامه به یک Provider در روز اول ساده است، اما با بزرگ‌شدن پروژه مشکلات آن ظاهر می‌شود: مدل حذف می‌شود، سهمیه رایگان تغییر می‌کند، Rate limit پاسخ‌گو نیست یا دسترسی منطقه‌ای دچار اختلال می‌شود.

معماری چند Provider قرار نیست همه درخواست‌ها را هم‌زمان به چند سرویس بفرستد. هدف این است که وابستگی کنترل شود و مسیر جایگزین قابل پیش‌بینی باشد.

### لایه‌بندی پیشنهادی

```text
Application
    ↓
LLM Gateway داخلی
    ↓
Routing Policy
    ↓
Provider A / Provider B
```

برنامه اصلی فقط با Gateway داخلی صحبت می‌کند. Gateway تنظیمات Provider، Timeout، Retry، Log و تبدیل پاسخ را مدیریت می‌کند.

### قرارداد واحد

یک مدل داده داخلی تعریف کنید:

```text
messages
temperature
max_output_tokens
tools
response_format
```

سپس هر Adapter این قرارداد را به فرمت Provider تبدیل کند. نباید فرض کرد همه سرویس‌های OpenAI-compatible تمام پارامترها را دقیقاً یکسان پشتیبانی می‌کنند.

### چه زمانی Fallback فعال شود؟

Fallback برای خطاهای موقت مناسب است:

- Timeout؛
- خطای ظرفیت؛
- 429 پس از عبور از سیاست Retry؛
- خطای 5xx؛
- مدل موقتاً در دسترس نیست.

اما برای این خطاها بهتر است خودکار Provider عوض نشود:

- 401 یا کلید نامعتبر؛
- ورودی ممنوع یا ناسازگار؛
- خطای برنامه‌نویسی؛
- درخواست حاوی داده‌ای که اجازه ارسال به Provider دوم را ندارد.

در غیر این صورت خطای تنظیمات پنهان می‌شود یا داده بدون تصمیم آگاهانه به سرویس دیگری می‌رود.

### Routing بر اساس کاربرد

همه درخواست‌ها نیاز به قوی‌ترین مدل ندارند. می‌توان Policy ساده‌ای داشت:

- خلاصه‌سازی کوتاه → مدل سریع و کم‌مصرف؛
- Reasoning پیچیده → مدل توانمندتر؛
- Embedding → Endpoint تخصصی؛
- خروجی JSON → Provider دارای Structured output معتبر؛
- درخواست حساس → Provider تأییدشده از نظر سیاست داده.

### سهمیه رایگان را Budget فرض نکنید

Free Tier برای شروع و تست مفید است، اما برای SLA تجاری تضمین ایجاد نمی‌کند. این موارد باید اندازه‌گیری شوند:

- تعداد درخواست موفق؛
- توکن ورودی و خروجی؛
- نرخ 429؛
- Latency؛
- درصد استفاده از Fallback؛
- هزینه احتمالی پس از عبور از سهمیه.

اگر Fallback همیشه فعال می‌شود، مشکل اصلی را با مسیر دوم پنهان کرده‌اید.

### جلوگیری از Loop و مصرف دوبرابر

برای هر درخواست یک شناسه ایجاد کنید و حداکثر تعداد تلاش را محدود سازید. مثال:

```text
Provider A: یک تلاش + یک Retry
Provider B: یک تلاش
سپس خطای کنترل‌شده
```

در عملیات غیر Idempotent مراقب باشید؛ ممکن است درخواست اول در Provider پردازش شده باشد اما پاسخ به برنامه نرسیده باشد.

### Observability بدون افشای داده

Log مفید شامل این موارد است:

- Provider و Model ID؛
- Status code؛
- Latency؛
- تعداد توکن؛
- نوع خطا؛
- فعال‌شدن Fallback.

Prompt کامل، API Key و اطلاعات شخصی نباید به‌صورت پیش‌فرض وارد Log شوند.

### انتخاب Providerهای اولیه

برای هر گزینه این فیلدها را کنار هم بگذارید:

- Free Tier یا Credit؛
- نیاز به کارت؛
- OpenAI compatibility؛
- Rate limits؛
- قابلیت موردنیاز؛
- وضعیت دسترسی ایران؛
- تاریخ آخرین بررسی؛
- سیاست داده و شرایط استفاده.

کاتالوگ **Awesome Free LLM APIs IR** این داده‌ها را برای Providerهای مختلف به شکل فارسی و ماشین‌خوان ارائه می‌کند و می‌تواند نقطه شروع ساخت ماتریس انتخاب باشد.
