# روش‌شناسی گردآوری و راستی‌آزمایی

## اصل پایه

چهار ادعای مستقل بررسی می‌شوند:

1. آیا سرویس API عمومی دارد؟
2. آیا سهمیه‌ای واقعاً بدون پرداخت ارائه می‌کند؟
3. آیا ثبت‌نام، صدور کلید و درخواست احراز هویت‌شده از مسیر مستقیم ایران کار می‌کند؟
4. آیا همان Credential، Model، Endpoint و Payload از مسیر کنترل مستقیم خارجی نتیجه متفاوتی می‌دهد؟

موفق بودن یک مرحله، مراحل دیگر را ثابت نمی‌کند. HTTP 200 روی `/models`، پاسخ 401، بازشدن وب‌سایت یا موفقیت مسیر خارجی به‌تنهایی اثبات استنتاج از ایران نیست.

## نوع سرویس

هر رکورد با `service_type` مشخص می‌کند که مدل‌ساز رسمی، Gateway رسمی یا Gateway اجتماعی است. ابزارهایی که Cookie، HAR یا Session مرورگر را به API محلی تبدیل می‌کنند Provider رسمی محسوب نمی‌شوند. تعریف کامل و قواعد نمایش در [نوع سرویس‌ها](SERVICE_TYPES.fa.md) آمده است.

## منابع سهمیه

اولویت شواهد به ترتیب زیر است:

1. مستندات رسمی قیمت و Rate Limit
2. داشبورد رسمی حساب، بدون انتشار اطلاعات خصوصی
3. پاسخ HTTP و Headerهای نرخ در تست واقعی
4. گزارش جامعه با مراحل بازتولیدپذیر

مطالب وبلاگی و شبکه‌های اجتماعی به‌تنهایی برای ثبت عدد سهمیه کافی نیستند.

## تست دسترسی ایران

تست معتبر باید حداقل شامل تاریخ UTC، نتیجهٔ ثبت‌نام، نتیجهٔ صدور کلید، نتیجهٔ یک درخواست کم‌هزینه، کشور، Route مستقیم، ASN عمومی و کد وضعیت یا کلاس خطای sanitized باشد. شهر اختیاری است.

برای نتیجه‌گیری منطقه‌ای، دو مسیر باید دقیقاً از Credential، Model ID، Endpoint، Payload، Timeout و Output limit یکسان استفاده کنند:

- مسیر مستقیم ایران؛
- مسیر کنترل مستقیم خارجی و مجاز.

اگر Credential در مسیر کنترل نیز معتبر نباشد، شکست ایران به‌عنوان Geo-block ثبت نمی‌شود. خطای حساب، Billing، سهمیه، Model ID و فعال‌سازی باید پیش از نتیجه‌گیری شبکه‌ای تفکیک شوند.

نتیجهٔ یک ASN به کل ایران تعمیم داده نمی‌شود. اگر نتایج شبکه‌ها متفاوت باشند، وضعیت `intermittent` یا توضیح محدود به همان Route ثبت می‌شود.

## مرزهای انتشار Evidence

موارد مجاز:

- کشور؛
- ASN عمومی؛
- `route=direct` یا `route=vpn`؛
- Latency گرد‌شده؛
- HTTP status؛
- کلاس خطای sanitized؛
- Model ID عمومی؛
- موفقیت یا شکست استنتاج.

موارد ممنوع:

- هر IPv4 یا IPv6 کامل در Evidence؛
- SSH target، Username یا جزئیات Host؛
- API key، Token، Prefix قابل‌شناسایی، Suffix، Hash یا مقدار ظاهراً Redacted؛
- Email، شماره تلفن، Header یا Response body دارای دادهٔ حساب؛
- شناسهٔ Account، Team، Organization، Tenant، Workspace یا Customer؛
- مسیر محلی فایل، نام کاربری سیستم یا جزئیات زیرساخت داخلی.

Gateهای `privacy:test` و `privacy:evidence:test` باید Source Provider JSON، Backlog، خروجی‌های Generated و محتوای عمومی مرتبط را بررسی کنند.

## رخداد افشای Credential

اگر هر بخش قابل‌شناسایی از Credential وارد Git history، Issue، PR، Log یا Artifact عمومی شد:

1. Credential فوراً در Dashboard Provider لغو یا Rotate شود؛
2. Branch آلوده Merge نشود؛
3. بازسازی از یک Base پاک انجام شود و Commit آلوده Cherry-pick نشود؛
4. تمام Diff و History شاخهٔ جایگزین با Privacy Gate بررسی شود؛
5. فقط تأیید انجام Rotation ثبت شود؛ مقدار قدیم، جدید، Prefix، Suffix و Hash هرگز منتشر نشود.

پاک‌کردن متن از آخرین Commit به‌تنهایی Credential افشاشده را امن نمی‌کند.

## تست با VPN

VPN فقط زمانی استفاده می‌شود که هدف ثبت وضعیت `verified_working_vpn` یا `direct_blocked_vpn_working` باشد. نتیجهٔ VPN هرگز به‌عنوان دسترسی مستقیم از ایران ثبت نمی‌شود. گزارش باید Route، کشور خروجی و نام عمومی سرویس VPN را بدون شناسهٔ حساب یا IP کامل ثبت کند.

برای کنترل علمی شکست مسیر مستقیم، سرور کنترل مستقیم خارجی بر VPN ترجیح دارد؛ زیرا Route و شرایط آزمون پایدارتر و قابل‌مقایسه‌تر است.

## تازگی داده

هر فایل `last_checked` و `stale_after_days` دارد. منقضی شدن به معنی غلط بودن داده نیست؛ یعنی پیش از اتکا باید دوباره بررسی شود. پیش‌فرض سهمیه‌ها ۳۰ روز است.

## امتیازدهی مشاور (Advisor)

بخش مشاور در وب‌سایت، Providerها را بر اساس اولویت و کاربرد رتبه‌بندی می‌کند. امتیاز از جمع وزن‌های زیر محاسبه می‌شود:

| شرط | امتیاز پایه | امتیاز در اولویت خاص |
|---|---|---|
| تطابق capability با usecase | ۳۵+ | — |
| OpenAI-compatible | ۱۰+ | ۲۸+ (اولویت `openai-compatible`) |
| سهمیهٔ دائمی (`permanent_allowance`) | ۱۸+ | — |
| مدل رایگان (`free_models`) | ۱۴+ | — |
| بدون نیاز به کارت بانکی | ۸+ | ۲۰+ (اولویت `low-friction`) |
| به‌روز بودن (`!isStale`) | ۸+ | ۲۰+ (اولویت `fresh`) |
| **دسترسی ایران (اولویت `iran-aware`)** | | |
| `verified_working` | ۳۰+ | — |
| `verified_working_vpn` | ۱۲+ | — |
| `officially_unsupported`, `verified_blocked`, `signup_blocked`, `account_activation_blocked` | ۳۰- | — |
| `unknown`, `intermittent` | ۰ | — |
| **دسترسی ایران (سایر اولویت‌ها)** | | |
| غیر از `unknown` | ۵+ | — |
| `officially_unsupported`, `verified_blocked` | ۱۲- | — |
| **نوع سرویس** | | |
| `community_gateway` | ۲۰- | — |
| `session_bridge` | ۳۰- | — |

اولویت `iran-aware` تضمین می‌کند که Providerهای مسدود یا پشتیبانی‌نشده از Providerهای نامشخص بالاتر نمایش داده نشوند. Providerهای `session_bridge` و `community_gateway` با وزن منفی در انتهای فهرست قرار می‌گیرند.

## محدودیت حقوقی

این پروژه روش دور زدن محدودیت جغرافیایی، تحریم، احراز هویت یا شرایط استفاده را آموزش نمی‌دهد. هدف فقط گزارش شفاف وضعیت قابل مشاهده است.
