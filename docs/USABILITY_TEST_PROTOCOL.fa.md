# پروتکل تست کاربردپذیری — Awesome Free LLM APIs IR

## هدف

این پروتکل بررسی می‌کند آیا تغییرات UI/UX واقعاً سه مسئله اصلی را حل کرده‌اند:

1. کاربر تازه‌وارد هدف سایت را سریع می‌فهمد؛
2. کاربر مسیر مناسب خود را بدون کمک پیدا می‌کند؛
3. برنامه‌نویس می‌تواند از انتخاب API به مستندات و اولین درخواست برسد.

هدف جلسه اثبات درست‌بودن طراحی نیست. پژوهشگر باید شکست، مکث، برداشت اشتباه و راه‌حل‌های غیرمنتظره کاربر را بدون دفاع از محصول ثبت کند.

## ترکیب نمونه

حداقل ۱۰ جلسه مستقل:

- ۵ کاربر عادی که تجربه عملی API ندارند؛
- ۲ برنامه‌نویس تازه‌کار؛
- ۳ برنامه‌نویس باتجربه.

در صورت امکان حداقل سه جلسه روی موبایل و سه جلسه روی دسکتاپ اجرا شود. شخصی که در توسعه پروژه نقش مستقیم داشته نباید در نمونه اصلی حساب شود.

## مرز حریم خصوصی

ثبت این موارد ممنوع است:

- نام کامل، ایمیل، شماره تلفن یا شناسه حساب؛
- IP کامل، Cookie یا Session؛
- API Key یا Authorization Header؛
- تصویر داشبورد خصوصی Provider؛
- اطلاعات پرداخت یا روش دورزدن محدودیت سرویس.

هر شرکت‌کننده با یک شناسه ناشناس مانند `OU-01` یا `DEV-03` ثبت شود.

## آماده‌سازی جلسه

- از نسخه Production و یک پنجره Private/Incognito استفاده شود.
- Cache یا وضعیت قبلی فرم‌ها پاک شود.
- نوع دستگاه، اندازه تقریبی نمایشگر و تاریخ جلسه ثبت شود.
- Moderator نباید قبل از تست، هدف سایت یا معنی API را توضیح دهد.
- ضبط صفحه یا صدا فقط با رضایت صریح شرکت‌کننده انجام شود؛ در غیر این صورت یادداشت ساختاریافته کافی است.

## تست پنج‌ثانیه‌ای

1. صفحه اصلی را پنج ثانیه نمایش بده.
2. صفحه را مخفی کن یا Tab را عوض کن.
3. بدون ارائه گزینه، بپرس:
   - «این سایت چه کاری انجام می‌دهد؟»
   - «فکر می‌کنی برای چه کسی ساخته شده؟»
   - «اگر بخواهی ادامه بدهی، اولین کاری که می‌کنی چیست؟»
4. اولین جمله کاربر را تقریباً عیناً ثبت کن.

### امتیاز فهم

- `0`: هدف را اشتباه فهمیده یا پاسخی مرتبط ندارد؛
- `1`: موضوع AI/API را فهمیده اما کار اصلی انتخاب و مقایسه را نفهمیده؛
- `2`: می‌گوید سایت برای پیدا کردن یا مقایسه API رایگان هوش مصنوعی است.

## سناریوهای کاربر عادی

### OU-1 — فهم API

**دستور:** «بدون استفاده از گوگل، بفهم API هوش مصنوعی چیست.»

معیار موفقیت:

- بخش توضیح ساده API را پیدا کند؛
- بتواند با کلمات خود بگوید API برنامه را به سرویس هوش مصنوعی وصل می‌کند.

### OU-2 — انتخاب مسیر

**دستور:** «می‌خواهی برای یک ربات پاسخ‌گو گزینه مناسب پیدا کنی؛ از هر مسیری که منطقی است شروع کن.»

معیار موفقیت:

- مسیر تازه‌کار یا API Finder را انتخاب کند؛
- بدون بازکردن تصادفی چند بخش، به پیشنهادها برسد.

### OU-3 — معنی وضعیت ایران

**دستور:** «بفهم آیا بازشدن سایت یک سرویس یعنی API آن از ایران حتماً کار می‌کند.»

معیار موفقیت:

- تشخیص دهد Reachability، Signup، Key issuance و Inference جدا هستند؛
- پاسخ «خیر» را با یک دلیل قابل‌فهم بیان کند.

## سناریوهای برنامه‌نویس

### DEV-1 — پیدا کردن Provider

**دستور:** «برای یک چت‌بات فارسی، یک گزینه بدون کارت بانکی پیدا کن و دلیل پیشنهاد را توضیح بده.»

معیار موفقیت:

- API Finder را پیدا کند؛
- سه سؤال اصلی را تکمیل یا تأیید کند؛
- یک نتیجه را بر اساس دلیل انسانی و وضعیت ثبت‌شده انتخاب کند؛
- امتیاز را با تضمین کیفیت اشتباه نگیرد.

### DEV-2 — بررسی شواهد و مستندات

**دستور:** «قبل از ثبت‌نام، منبع رسمی و وضعیت ایران گزینه انتخاب‌شده را بررسی کن.»

معیار موفقیت:

- صفحه Provider را باز کند؛
- تاریخ بررسی و منبع رسمی را پیدا کند؛
- وضعیت شبکه را با اجرای واقعی مدل یکی نداند.

### DEV-3 — اولین درخواست

**دستور:** «بدون ساخت حساب واقعی، مسیر آماده‌سازی اولین درخواست Node.js را پیدا کن و بگو چه سه مقدار باید جایگزین شوند.»

معیار موفقیت:

- Quick Start را پیدا کند؛
- نمونه Node.js را باز کند؛
- `LLM_API_KEY`، `LLM_BASE_URL` و `LLM_MODEL` را نام ببرد؛
- بفهمد کلید نباید در کد عمومی قرار گیرد.

### DEV-4 — رفع خطا

**دستور:** «فرض کن پاسخ 401 گرفته‌ای. بفهم این خطا چه چیزی را ثابت می‌کند و قدم بعدی چیست.»

معیار موفقیت:

- 401 را نشانه کلید مفقود/نامعتبر یا احراز هویت ناموفق بداند؛
- آن را اثبات موفقیت Inference تلقی نکند؛
- Header، کلید و Endpoint را به‌عنوان نقاط بررسی ذکر کند.

## رفتار Moderator

- متن سناریو را بخوان؛ راه‌حل را توضیح نده.
- در سکوت اجازه بده کاربر فکر کند.
- اگر کاربر سؤال پرسید، بگو: «همان کاری را انجام بده که اگر تنها بودی انجام می‌دادی.»
- فقط زمانی مداخله کن که کاربر کاملاً متوقف شده یا خطر افشای اطلاعات محرمانه وجود دارد.
- هر مداخله به‌معنی شکست انجام مستقل کار ثبت شود.
- گفته کاربر، محل کلیک اشتباه و لحظه مکث را ثبت کن؛ فقط نتیجه نهایی کافی نیست.

## داده‌های هر جلسه

- شناسه ناشناس؛
- Audience؛
- Device؛
- تاریخ؛
- امتیاز فهم ۰ تا ۲؛
- سناریوی اجراشده؛
- نتیجه: Success / Partial / Failed؛
- زمان انجام؛
- تعداد مداخلات Moderator؛
- اولین نقطه سردرگمی؛
- شدت: None / Low / Medium / High / Critical؛
- اطمینان کاربر از نتیجه، ۱ تا ۵؛
- نقل‌قول کوتاه و غیرشخصی؛
- Issue مرتبط در صورت نیاز.

## قواعد امتیازدهی

### Task Success

- `1`: کامل و بدون کمک؛
- `0.5`: کامل با مکث جدی یا یک مداخله؛
- `0`: ناقص، اشتباه یا نیازمند راهنمایی مستقیم.

### شدت مشکل

- `None`: بدون مکث معنادار؛
- `Low`: مکث کوتاه، بدون اثر بر نتیجه؛
- `Medium`: آزمون چند مسیر یا بازگشت، اما موفقیت مستقل؛
- `High`: نیاز به مداخله یا برداشت اشتباه مهم؛
- `Critical`: هدف محصول، رایگان‌بودن یا Evidence ایران به‌طور خطرناک اشتباه فهمیده شود.

## آستانه‌های موفقیت فاز

برای بسته‌شدن Issue اصلی UX باید حداقل این شرایط برقرار باشد:

- حداقل ۸۰٪ کاربران عادی امتیاز فهم `2` بگیرند؛
- حداقل ۸۰٪ کل شرکت‌کنندگان مسیر اصلی خود را بدون مداخله پیدا کنند؛
- حداقل ۸۰٪ توسعه‌دهندگان DEV-1 و DEV-3 را با Task Success برابر `1` انجام دهند؛
- Median زمان رسیدن از Homepage به نتایج Finder کمتر از ۲ دقیقه باشد؛
- هیچ مشکل Critical حل‌نشده درباره معنی Free Tier یا Iran Evidence باقی نماند؛
- Confidence میانه برای انتخاب نهایی حداقل ۴ از ۵ باشد.

## تحلیل و اولویت‌بندی

پس از هر پنج جلسه:

1. مشکلات مشابه را Cluster کن؛
2. تعداد وقوع، شدت و Journey را ثبت کن؛
3. اولویت را با ترکیب `Frequency × Severity × Core-task impact` تعیین کن؛
4. برای مشکل قابل‌اقدام Issue مستقل بساز؛
5. متن خام یا اطلاعات شخصی را در GitHub عمومی منتشر نکن؛
6. پس از اصلاح، همان سناریو را با شرکت‌کنندگان جدید دوباره اجرا کن.

## خروجی نهایی

گزارش فاز باید شامل این موارد باشد:

- تعداد جلسات به تفکیک Audience و Device؛
- نرخ فهم پنج‌ثانیه‌ای؛
- Task Success هر سناریو؛
- Median زمان؛
- Confidence میانه؛
- پنج مشکل پرتکرار؛
- مشکلات High/Critical؛
- Issueهای ایجادشده؛
- تصمیم‌های محصولی و مواردی که عمداً تغییر نکرده‌اند.
